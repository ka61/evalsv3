A Principles

The rules that drive the order

Build order is not arbitrary — it follows from a few principles. Each one is chosen to keep the project demoable, cheap, and safe at every step, so a half-finished platform is still a working one.

// 01

Thin slice first

Get the simplest eval running end-to-end (API model on Fargate → S3) before adding any GPU or Kubernetes. A working narrow path beats a broad scaffold.

// 02

Paved path from day one

CI, policy gates, and protected main exist before the first real resource. You never apply by hand, even on day one.

// 03

Cheap by default

Every capability ships behind a feature flag with a cheap default. Expensive switches (EKS, OpenSearch) stay off until needed.

// 04

Interface before breadth

Prove the GPU-provider interface on AWS first; only then add CoreWeave, RunPod, Lambda behind the same contract.

// 05

Two altitudes always

Every capability is observable account-wide and per-run from the start — logging and tagging are wired in, not retrofitted.

// 06

Demoable milestones

Each phase ends with something you can show: open a PR, run an eval, capture egress, read the cost. Progress is visible.

B Roadmap

Phase roadmap

Four phases over roughly eight weeks. Foundations and the simple path come first; the agentic path, then observability and the console, then breadth. Bars below show when each workstream is active; colour marks its phase.

C Dependencies

The dependency graph

Workstreams can't start in any order. Remote state comes before everything; networking and storage gate the executors; the sandbox gates the agentic path. The critical path runs straight down the middle — anything off it can be parallelised.

D Workstreams

Section-by-section implementation

Each workstream below produces one or more Terraform modules and ends at a testable state. They are ordered along the dependency graph; phase tags show when each runs. Every card states its objective, the steps, the acceptance criteria that close it, and the risks to watch.

P11 · Remote state & bootstrapmodule: backend

Objective. Establish the foundation every other module assumes: versioned, locked Terraform state and a clean account baseline. Nothing is applied by hand after this.

Create the state backend

S3 bucket (versioned, encrypted, public-access blocked) + DynamoDB lock table, bootstrapped once with a local backend then migrated.

Define environments

envs/dev and envs/prod with isolated state keys and per-env variables; account-level guardrails (tag policy, region pin).

Wire OIDC roles

GitHub OIDC provider + least-privilege deploy roles so CI can assume-role with no stored keys.

Acceptance criteria
  • terraform plan is clean from a fresh clone
  • State is remote, locked, and versioned
  • CI can apply via OIDC, no long-lived keys exist
Risks
  • Chicken-and-egg bootstrap of the backend — script it and document the one-time step
  • Over-broad deploy role — scope per environment
P12 · Networkingmodule: networking

Objective. A VPC that is cheap by default and audit-ready: private subnets, VPC endpoints instead of a NAT tax, and Flow Logs on from day one.

VPC + subnets

Multi-AZ private/public subnets, route tables, security groups with least-privilege defaults.

Kill the NAT tax

Gateway endpoint for S3 (free) and interface endpoints for ECR, CloudWatch, Secrets Manager; NAT only where genuinely needed.

Flow Logs

VPC Flow Logs to S3, partitioned by run — the base layer of network inspectability.

Acceptance criteria
  • Workloads reach S3/ECR with no NAT data charges
  • Flow Logs land in S3 and are queryable
  • Idle networking cost is near zero
Risks
  • Missing an endpoint silently re-introduces NAT cost — assert in a policy test
P13 · Storage, state store & secretsmodules: storage · state-store

Objective. Durable homes for logs/results, run metadata, and credentials — the data backbone for both eval paths.

S3 log/result buckets

Lifecycle rules to Infrequent-Access then Glacier; .eval transcripts archived here.

RDS Postgres

Small instance for run metadata; schema + migrations checked into the repo.

Secrets Manager

Model API keys stored here only — never in images or state; rotation policy defined.

Acceptance criteria
  • A written object expires per lifecycle policy
  • Migrations run idempotently in CI
  • No secret appears in state or image layers
Risks
  • RDS is always-on cost — size it small, document the stop option
P14 · Audit baselinemodule: audit

Objective. Make every action attributable before any workload runs — the safety floor of the platform.

CloudTrail

Account-wide trail to a write-once (object-lock) S3 bucket.

Config + GuardDuty

Config rules for drift/posture; GuardDuty findings surfaced to the dashboard later.

IRSA roles

Each workload assumes a least-privilege role; no shared keys.

Acceptance criteria
  • API calls appear in an immutable trail
  • Config flags an intentional drift in test
Risks
  • CloudTrail/Config volume cost — scope data events deliberately
P15 · Control plane API + queuemodule: control-plane

Objective. The brain: accept eval specs, enqueue them, and route each to the right executor by {sandbox, model, provider}.

Acceptance criteria
  • A submitted spec is validated, tagged, and enqueued
  • Failures land in a DLQ, not a black hole
  • Router picks the correct executor by spec
Risks
  • Router logic creep — keep placement rules declarative
P16 · Fargate executor (simple path)module: fargate-executor

Objective. Run model-graded evals end-to-end with no GPU and no cluster — the first demoable milestone.

Task definition

Container running inspect eval, API key from Secrets Manager, scale-to-zero.

Logs & transcripts

Structured logs to CloudWatch/S3; .eval transcript archived to S3.

First dashboard read

A minimal pane lists completed runs from metadata.

Acceptance criteria
  • A model-graded eval runs from a submitted spec
  • Its .eval log is readable in S3
  • Cost for an idle month is a few dollars
Risks
  • Model-provider rate limits — add backoff and concurrency caps
P17 · CI/CD & paved-path repomodule: cicd

Objective. Make Git the only write path before any real resource exists. This runs from day one and stays ongoing.

PR pipeline

fmt · validate · tflint · checkov · plan posted to the PR; OPA/Conftest policy gate.

Apply pipeline

OIDC apply to dev on merge; prod behind required reviewers + wait timer.

Protections + drift

Protected main, CODEOWNERS, secret scanning; scheduled drift detection.

Acceptance criteria
  • An unreviewed change cannot reach the account
  • A policy violation blocks merge
  • Drift between live and main raises an alert
Risks
  • Gate fatigue — keep policies few and meaningful
P28 · EKS + AISI sandbox (agentic path)modules: eks · eks-sandbox-config

Objective. Run sandboxed agentic evals safely — the highest-risk, highest-value workstream. Untrusted code runs only inside a contained, egress-controlled pod.

Acceptance criteria
  • A sandboxed agentic eval runs in-cluster
  • An off-allow-list egress is blocked and recorded
  • GPU nodes scale to zero when idle
Risks
  • Highest-risk module. Confirm current AISI k8s-sandbox requirements (CNI, NetworkPolicy, versions) before building
  • Sandbox escape — treat isolation as security-critical
P39 · Observability & FinOps tiersmodules: observability · logging · finops

Objective. See operational and eval health together, and make spend attributable — all behind cheap-by-default feature flags.

Metrics & traces

Prometheus/Grafana + OTel spans (submit→score→archive); DCGM only when GPUs are up.

Tiered logging

Tier-0 S3 + CW Infrequent-Access default; OpenSearch flagged off.

Cost attribution

Cost-allocation tags, budgets/alarms, per-run cost stored beside metadata.

Acceptance criteria
  • A run's cost is visible the moment it finishes
  • Default tiers keep idle spend near baseline
  • A budget breach raises an alarm
Risks
  • Metric cardinality blow-up — watch label explosion
P310 · Dashboard (the console)module: dashboard

Objective. One surface for passive inspection and active, audited control — from account health to a single sample.

Acceptance criteria
  • Drill from account → run → sample without switching tools
  • Active actions (pause/kill/requeue) write audit events
Risks
  • Active actions need real authz — see production gaps
P411 · Multi-provider GPUmodule: gpu-providers/*

Objective. Make GPU work portable across providers behind one interface — only after the AWS adapter is proven.

Common interface

One gpu_provider contract; AWS adapter first as the reference.

Add adapters

CoreWeave (join K8s), RunPod (burst), Lambda (reserved Docker host) behind the same spec.

Honest portability

Router records which network guarantees applied per provider so the audit trail stays truthful.

Acceptance criteria
  • A job lands on the cheapest qualifying provider unchanged
  • Per-run audit records the provider and its guarantees
Risks
  • Host-based providers offer weaker egress control — record the tradeoff, don't hide it
E Production gaps

What's missing for an org like Apollo Research

The reference build is a strong, honest platform — but a safety-evals organisation running pre-deployment evaluations of frontier models (deception, scheming, dangerous-capability work) under agreements with labs and governments has a higher bar. The design is deliberately scoped; below is what production at that bar additionally requires. The chart shows where the reference build is already strong versus where the gap is largest.

// GAP 01

Identity, SSO & RBAC

The console audits actions but assumes a single operator. Production needs SSO (Okta/SAML/OIDC), role-based access control, and per-team scoping so "who can kill a run" or "who can read these results" is enforced, not just logged.

// GAP 02

Secure model & weight access

Pre-deployment access to frontier models comes with strict security terms. Self-hosted weights need encrypted, access-controlled, no-egress handling (KMS, isolated nodes, possibly confidential computing) and attested compliance with each lab's requirements.

// GAP 03

Multi-tenancy & quotas

Teams and projects with isolated state, GPU quotas, fair scheduling, and budget ceilings — so one campaign can't starve another or blow the org budget.

// GAP 04

Sandbox hardening

For genuinely dangerous-capability work, container isolation isn't enough. Add stronger runtimes (gVisor / Kata / Firecracker), seccomp/AppArmor, runtime threat detection (Falco), and a documented escape threat model.

// GAP 05

Data governance & infohazards

Results of dangerous-capability evals can themselves be sensitive. Need data classification, tiered access, encryption, retention/right-to-delete, residency controls, and an infohazard-handling policy baked into storage.

// GAP 06

HA, DR & SLOs

Multi-AZ RDS with failover, tested backups and disaster-recovery runbooks, defined SLOs/error budgets, and an incident-management/on-call process.

// GAP 07

Scale & rate governance

Large parallel campaigns need queue backpressure, autoscaling caps, and coordinated rate-limiting against model-provider APIs, plus hard cost guardrails.

// GAP 08

Compliance & assurance

SOC 2 / ISO 27001 posture, change-management evidence, periodic access reviews, vendor risk, and penetration testing — table stakes for handling labs' and governments' trust.

// GAP 09

First-class custom evals

The spec scopes out novel evals. Apollo authors bespoke deception/scheming evals — so production needs a reviewed authoring workflow, elicitation tooling, and human-baseline/uplift infrastructure.

// GAP 10

Result integrity & provenance

Strengthen the already-good audit trail into signed, tamper-evident artifacts with reproducibility attestation and statistical rigor (multiple seeds, variance, significance) before results inform high-stakes decisions.

Bottom line — the reference design nails the infrastructure story: isolation, observability, cost discipline, GitOps. The production delta for an org like Apollo Research is mostly the governance story — identity, secure model access, multi-tenancy, compliance, and eval-authoring rigor. None of it invalidates the build order above; it's a Phase 5+ track layered on the same foundations.
