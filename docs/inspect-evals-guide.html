01 Mental model

The anatomy of an eval

Everything in Inspect is organised around a Task: a dataset of samples, a solver that produces an answer for each, and a scorer that grades it. Hold this shape in your head and the rest of the API falls into place.

// DATASET

What to ask

A list of Samples, each an input and (usually) a target. Loaded from files or the Hugging Face Hub.

// SOLVER

How to answer

A pipeline that turns a sample into a model response — from a bare generate() to a tool-using agent.

// SCORER

How to grade

Turns each response into a score, aggregated into metrics like accuracy with a standard error.

02 Setup

Install, configure, and run your first eval

Inspect is a Python package. You install it, expose your model provider's API key as an environment variable, write a task file, and run it from the command line. The log viewer then shows what happened.

Install the package

pip install inspect-ai (plus a provider SDK, e.g. pip install openai or anthropic).

Set your API key

Export the provider key, e.g. export OPENAI_API_KEY=sk-… — Inspect reads provider keys from the environment.

Write a task

A .py file with a @task function returning a Task.

Run it

inspect eval task.py --model openrouter/openai/gpt-5.4 — runs the dataset and writes a log.

View results

inspect view opens the interactive log viewer in your browser.

A minimal project — capitals/

# capitals/task.py
from inspect_ai import Task, task
from inspect_ai.dataset import example_dataset
from inspect_ai.scorer import model_graded_fact
from inspect_ai.solver import generate, system_message

@task
def capitals():
    return Task(
        dataset=example_dataset("capitals"),
        solver=[
            system_message("Answer with the city name only."),
            generate(),
        ],
        scorer=model_graded_fact(),
    )
# run from the shell
inspect eval capitals/task.py --model openrouter/openai/gpt-5.4
inspect eval capitals/task.py --model openrouter/anthropic/claude-opus-4.8 --limit 10
inspect view   # browse the .eval logs
Project layout — keep tasks, any datasets, and (for agentic evals) a Dockerfile/compose.yaml together in one directory. Logs land in ./logs by default, configurable via INSPECT_LOG_DIR.
03 Dataset

The dataset: samples in, comparably

A dataset is a sequence of Sample objects. Inspect loads them from JSON/CSV files, the Hugging Face Hub, or builds them in code — and lets you map arbitrary record fields onto the Sample shape.

Sample fieldMeaning
inputThe prompt — a string or a list of chat messages
targetThe correct answer or grading reference
choicesOptions, for multiple-choice tasks
idA stable identifier for the sample
metadataArbitrary extra fields (difficulty, category, …)
filesFiles to mount into the sandbox for this sample
setupA setup script run in the sandbox before the solver
sandboxPer-sample sandbox override (e.g. a specific image)

Loaders — file, Hub, and field mapping

from inspect_ai.dataset import json_dataset, csv_dataset, hf_dataset, Sample

# 1) straight from a JSONL file
ds = json_dataset("questions.jsonl")

# 2) from the Hugging Face Hub, mapping record fields → Sample fields
def record_to_sample(record):
    return Sample(
        input=record["question"],
        target=record["answer"],
        metadata={"category": record["category"]},
    )

ds = hf_dataset("gsm8k", split="test", sample_fields=record_to_sample)

# 3) built in code
ds = [Sample(input="2+2?", target="4")]
Where eval data comes from — most public benchmarks (GSM8K, MMLU, GPQA, HumanEval, SWE-bench, …) live on the Hugging Face Hub. The inspect_evals package already wraps many of them, so you often don't write the loader yourself.

04 Solver

The solver: from one call to an agent

A solver transforms the sample into a model answer. Solvers are composable steps run in order on a shared TaskState. The simplest is a single generate(); the most complex is a full tool-using agent that loops until it submits an answer.

Built-in solvers you'll reach for

SolverUse
generate()Call the model on the current state (the workhorse)
system_message() / prompt_template()Inject a system prompt or reshape the user prompt
multiple_choice()Format choices and parse the selected option
use_tools()Make a set of tools available to the model
react() (agent)The recommended agent — a ReAct tool loop with a submit() tool and attempts
chain() / a listCompose multiple solvers in sequence
self_critique()Have the model critique and revise its answer

The agent module — react() is the default

Modern Inspect centres agents on the inspect_ai.agent module. The general-purpose react() agent (ReAct: reason + act) is the recommended baseline — it runs a tool loop until the model calls a built-in submit() tool, supports multiple scored attempts, refusal retries, and context compaction for long runs. (The older basic_agent() solver still works, but react() supersedes it.) An agent can be used as a task solver, run directly with run(), exposed to a model as a tool with as_tool(), or composed into multi-agent systems via handoff().

from inspect_ai.agent import react
from inspect_ai.tool import bash, text_editor

# the modern, recommended agent — ReAct tool loop with a submit() tool
agent = react(
    prompt="You are a careful software engineer.",
    tools=[bash(timeout=180), text_editor()],
    attempts=3,          # retry on an incorrect submission (graded by the task scorer)
)
# use it anywhere a solver is accepted:
# Task(..., solver=agent)
Beyond react — Inspect also ships a Deep Agent (long-horizon tasks with sub-agent delegation, memory, planning), the Human Agent for human baselining, an Agent Bridge to run agents from OpenAI Agents SDK / LangChain / Pydantic AI, and Inspect SWE to drive Claude Code and Codex CLI as agents.
05 Tools

Tools and the agent loop

Tools are what turn a chat model into an agent. Inspect ships a rich set of standard tools, supports any MCP server, and lets you define your own with a decorator. When a tool executes code, it runs inside the sandbox (next section) — never on your host.

Standard tools

ToolWhat it does
bash() · python()Execute arbitrary shell / Python code in the sandbox
bash_session()A stateful bash shell that keeps state across calls
text_editor()View, create, and edit text files (the standard coding tool)
web_search()Run and summarise web searches (built-in or external provider)
web_browser()A headless Chromium browser — navigate, click, type, history
computer()A full desktop via screenshots with mouse/keyboard control
code_execution()Provider-side sandboxed Python (runs in the model vendor's infra)
think()An explicit extra reasoning step before answering
memory() · update_plan() · skill()Agentic helpers: persistent memory, plan tracking, skill specs
// MCP

MCP servers

Any of the hundreds of Model Context Protocol servers (filesystem, git, databases, browsing, …) can be wired in as tools — Inspect has first-class MCP support.

// CUSTOM

Your own tools

Decorate an async execute function with @tool; typed args and the docstring become the schema the model sees. Inside, call sandbox().exec(...) to act in the isolated environment.

from inspect_ai.tool import tool
from inspect_ai.util import sandbox

@tool
def grep():
    async def execute(pattern: str, path: str):
        """Search files for a pattern.

        Args:
            pattern: regex to search for
            path: directory to search
        """
        result = await sandbox().exec(["grep", "-rn", pattern, path])
        return result.stdout
    return execute
Safety — every tool that runs model-authored code must execute in the sandbox, with an egress allow-list and resource limits. Treat tool output as untrusted; treat tool actions as the thing you audit.
06 Scorer

The scorer and metrics

A scorer compares the model's answer to the target and emits a score with an explanation. Scores aggregate into metrics — usually accuracy with a standard error. Some scorers are deterministic; others ask a model to judge.

ScorerHow it grades
match() / includes()Exact or substring match against the target
pattern()Regex extraction then compare
choice()For multiple-choice — did it pick the right option
answer()Parse a structured "ANSWER: …" line
model_graded_qa() / model_graded_fact()A grader model judges against a rubric — for open-ended answers
custom @scorerRun a unit test in the sandbox, check a file, etc.
from inspect_ai.scorer import scorer, accuracy, stderr, Score, Target
from inspect_ai.solver import TaskState
from inspect_ai.util import sandbox

@scorer(metrics=[accuracy(), stderr()])
def tests_pass():
    async def score(state: TaskState, target: Target):
        # run the project's test suite inside the sandbox
        r = await sandbox().exec(["pytest", "-q"])
        return Score(value="C" if r.success else "I", explanation=r.stdout)
    return score
Epochs & reducers — run each sample multiple times (--epochs 5) and reduce the repeats (mean, or pass@k) to measure reliability, not just a single lucky run. Always report the standard error alongside accuracy.
07 Sandbox

Sandboxing & Docker

When an eval lets the model run code, that code must be isolated from your machine and the network. Inspect's sandbox abstraction provides a disposable environment per sample. The default provider is Docker (via a compose.yaml); for production isolation at scale there is a Kubernetes provider. Tools like bash() and python(), and sandbox().exec() in your own code, run inside this environment.

Declaring the sandbox — task + compose.yaml + Dockerfile

# task.py — attach a sandbox to the task
Task(
    dataset=ds,
    solver=react(tools=[bash(), python()]),    # react agent
    scorer=tests_pass(),
    sandbox=("docker", "compose.yaml"),       # provider, config
)
# compose.yaml — the default sandbox service
services:
  default:
    build: .                       # build from Dockerfile in this dir
    init: true                     # proper PID 1 / signal handling
    command: tail -f /dev/null      # keep the container alive for the run
    cpus: 1.0
    mem_limit: 0.5gb
    network_mode: none             # no network unless the eval needs it
You may not need a compose file — with no config, Inspect builds the sandbox from its standard aisiuk/inspect-tool-support image; with just a Dockerfile it builds your image; with a compose.yaml it uses that. Auto-generated compose files restrict internet access by default, so add your own compose.yaml if a task needs egress.
# Dockerfile — the environment the model acts in
FROM python:3.12-slim
RUN pip install --no-cache-dir pytest numpy pandas
WORKDIR /workspace
# sample files are mounted at run time; keep the image small & pinned

Sandbox providers

Two are built in; the rest install as small packages. Docker-compatible providers accept the same Dockerfile / compose.yaml, so you can develop locally on Docker and scale out on k8s or a cloud sandbox without rewriting the eval.

ProviderPackageUse for
dockerbuilt-inLocal dev & CI — the default
localbuilt-inNo isolation — runs on the host (use with care)
k8sinspect-k8s-sandboxScale across a cluster; Cilium NetworkPolicy egress control
daytona · modalinspect-sandboxesManaged cloud sandboxes
ec2 · proxmoxAISI packagesFull VMs (when a container isn't enough)
// MULTI-SERVICE

More than one container

A compose.yaml can define several services — e.g. an agent's default box plus a victim server on an isolated network. Reach the second with sandbox("victim"); place files with a victim: prefix.

// CLEANUP

Lifecycle

Containers are created and destroyed per sample. Run with --no-sandbox-cleanup to keep them for debugging, then inspect sandbox cleanup docker to remove them.

Docker details that matter — install Docker Engine ≥ 24.0.7; pin base images and deps for reproducibility; keep images small for fast per-sample startup; include init: true and command: tail -f /dev/null; bound runaway agents with cpus/mem_limit; for >30 concurrent containers, widen Docker's default address pools.
08 Models

Models & providers

The --model flag (or get_model()) selects a provider and model with a provider/name string. Hosted APIs need only a key; self-hosted open-weight models need GPUs and a serving engine like vLLM.

CategoryProviders (provider/model)
Lab APIsopenai/ · anthropic/ · google/ · grok/ · mistral/ · deepseek/ · perplexity/
Cloud APIsbedrock/ · sagemaker/ · azureai/
Open (hosted)groq/ · together/ · fireworks/ · cloudflare/ · hf/ (Inference) · sambanova/
Open (local)hf/ · vllm/ · sglang/ · ollama/ · llama-cpp-python/ · transformer-lens/
Anything elseopenai-api/ (any OpenAI-compatible endpoint) · openrouter/

Examples: openrouter/openai/gpt-5.4-mini, openrouter/anthropic/claude-opus-4.8, openrouter/google/gemini-3.5-flash, vllm/meta-llama/Llama-3.1-8B-Instruct. Set a default with INSPECT_EVAL_MODEL, or pass --model none for evals that call models internally.

# generation config & connection limits
inspect eval task.py --model openrouter/anthropic/claude-opus-4.8 \
  --temperature 0.0 --max-tokens 2048 \
  --max-connections 20      # parallel requests to the provider

# self-hosted open-weight model via vLLM
inspect eval task.py --model vllm/meta-llama/Llama-3.1-8B-Instruct

Model roles — name the models in your eval

Rather than hard-coding model names, give them roles (e.g. grader, red_team, blue_team) and bind real models at run time. Built-in model-graded scorers look for the grader role by default, so you can swap the grader without touching code.

# in the task, or override on the CLI / in eval()
Task(..., model_roles={"grader": "openrouter/openai/gpt-5.4"})

# CLI: vary the grader (or red/blue teams) per run
inspect eval math.py --model openrouter/anthropic/claude-opus-4.8 \
  --model-role grader=openrouter/google/gemini-3.5-flash

GPU sizing when you self-host the model

If the eval drives an open-weight model you serve yourself, you need enough GPU memory for the weights plus the KV cache. Rough guidance (BF16 ≈ 2 GB per billion params; quantization roughly halves it):

Model sizeApprox. weights (BF16)Typical serving GPU
7–8B~16 GB1× L4/A10G (quantized) or 1× A100-40
13B~26 GB1× A100-40/80
30–34B~65 GB1× A100-80 / H100-80 (or 2× smaller)
70B~140 GB2× A100-80 / 2× H100, or 1× with quantization
>100B (MoE/dense)200 GB+Multi-GPU H100/H200 with tensor parallelism
Most evals don't need a GPU — when you test a hosted API model, there is no GPU in the loop at all. GPUs enter only when you self-host the model under test, or when an agentic task's sandbox itself needs one.
09 Scale

Running at scale

A real eval is hundreds or thousands of samples, often repeated across epochs. Inspect runs samples concurrently, bounded by connection and subprocess limits, and can run whole suites as an eval-set with retries.

FlagEffect
--limit N / 10-20Run a maximum (or a range) of samples — fast iteration
--epochs NRepeat each sample N times for reliability
--epochs-reducerHow to reduce epochs: mean, median, max, at_least_{n}, pass_at_{k}
--max-connections NCap parallel model requests (rate-limit safety; default 10)
--adaptive-connectionsAuto-tune the connection count for you
--max-samples NSamples in parallel per task (defaults to --max-connections)
--max-sandboxes NConcurrent containers (default 2 × CPU count)
--max-subprocesses N · --max-tasks NCap subprocess work / tasks run in parallel
--fail-on-error · --retry-on-errorTolerate a proportion of sample errors / retry failed samples
--time-limit · --token-limit · --message-limit · --cost-limitPer-sample budgets that stop runaway agents
-T key=value · -S key=valuePass arguments into the @task / solver
--log-dir · --sandboxWhere logs go (local or s3://) / set the sandbox provider
inspect eval-setRun many tasks as a set, with automatic retries of failures
10 Logs

Logs and the viewer

Every run writes an .eval log — a self-contained record of every sample's messages, tool calls, scores, and the run's full configuration. It is the authoritative artifact: reproducible, inspectable, and archivable.

// THE FILE

The .eval log

Captures inputs, every message and tool call, model + provider, scores and metrics, and the exact config (incl. seed and git context). One file per run.

// THE VIEWER

inspect view

An interactive browser UI to step through samples and transcripts. inspect log and the Python API read logs programmatically for analysis.

inspect view                      # browse logs in ./logs
inspect view --log-dir s3://bucket/evals   # or a remote store
inspect score logs/2026-...eval  --scorer model_graded_qa  # re-score later

12 Environment

Setting up a local environment

The fastest, most reliable local setup is: a modern Python environment manager, the inspect-ai package, a provider key in a .env file, and Docker for sandboxed evals. Inspect requires Python ≥ 3.10. Below are the common options ranked by ease, then a step-by-step.

Option A — pick a Python environment manager

OptionWhySetup
uv (recommended)Fastest installs, reproducible lockfile, one tool for venv + packagesuv venv && uv pip install inspect-ai
venv + pipBuilt into Python, zero extra tooling — the safe defaultpython -m venv .venv && pip install inspect-ai
conda / mambaIf you already live in conda or need non-Python depsconda create -n evals python=3.12 && pip install inspect-ai

Step by step (uv path)

Create & activate an env

uv venv then source .venv/bin/activate (or python -m venv .venv for the pip path).

Install Inspect

uv pip install inspect-ai openai anthropic — the framework plus the provider SDKs you'll use.

Add a .env file

Put your keys and defaults there; Inspect (and the VS Code extension) load it automatically.

Install Docker

Docker Desktop (macOS/Windows) or Docker Engine (Linux) — needed only for sandboxed evals.

Smoke-test

Run a built-in eval against a cheap model with --limit 5, then inspect view.

14 Quickstart

Running the Docker-based eval, step by step

This is the concrete path from nothing to a sandboxed run. Inspect auto-discovers any Dockerfile and compose.yaml in the task directory, builds the image, and creates a fresh container per sample. Here's exactly what to do and what happens.

Confirm Docker is running

docker run --rm hello-world should succeed before you start.

Create the task directory

Put task.py, a Dockerfile, and a compose.yaml together — Inspect discovers the latter two automatically.

(Optional) pre-build the image

docker compose build to surface build errors early; otherwise Inspect builds on first run.

Run with the sandbox

inspect eval task.py --model … --limit 5; add --sandbox docker if not set in the task.

Watch & debug

docker ps shows live sandboxes; docker exec -it <id> bash opens a shell to inspect one.

Review

inspect view — every tool call the agent made is in the transcript.

# the commands, start to finish
mkdir myeval && cd myeval
# ... add task.py, Dockerfile, compose.yaml ...
docker compose build                         # optional: build ahead of time
inspect eval task.py --model openrouter/openai/gpt-5.4 --limit 5
docker ps                                     # see sandboxes while running
inspect view
# control concurrency & sandbox behaviour
inspect eval task.py --model openrouter/openai/gpt-5.4 \
  --max-sandboxes 8       # cap simultaneous containers \
  --max-connections 10    # cap model API parallelism \
  --epochs 3              # repeat each sample
// OPTIONS

Network & limits

Default network_mode: none for full isolation; switch to an allow-list only when a task needs egress. Set cpus and mem_limit in compose.yaml to bound a runaway agent.

// GOTCHAS

Common fixes

Tag local images with x-local: true so they aren't pulled; keep images small for fast startup; if a build fails, run docker compose build to see the full error; debug with --no-sandbox-cleanup then tidy up with inspect sandbox cleanup docker.

15 Cloud

Running in the cloud

Local is perfect for development; the cloud is for scale (thousands of samples, many parallel sandboxes), for GPUs (self-hosting the model under test), and for stronger isolation. There are four common paths, from a single VM to a managed platform — pick by how much scale and isolation you need.

OptionBest forIsolationEffort
Single cloud VM (EC2/GCE) + DockerBigger runs than a laptop; still simplePer-sample containers on one hostLow
Kubernetes + inspect_k8s_sandboxLarge-scale agentic evals across many nodesPer-pod, with Cilium NetworkPolicy egressMedium–High
Managed (e.g. METR Hawk)Turnkey: YAML in, results UI outIsolated k8s pods, managed for youLow (operationally)
Self-hosted model on GPU nodesOpen-weight models under testvLLM on GPU + sandbox separatelyMedium

Path 1 — a single cloud VM (easiest)

Launch a VM

An EC2/GCE instance with Docker installed (a GPU instance only if you self-host the model).

Install Inspect

pip install inspect-ai, set keys, copy your task directory over.

Run, logging to S3

inspect eval … --log-dir s3://bucket/evals so results persist beyond the VM.

# on the VM — run and persist logs to S3
export INSPECT_LOG_DIR=s3://my-evals-bucket/run-2026-06
inspect eval task.py --model openrouter/anthropic/claude-opus-4.8 \
  --max-connections 30 --max-sandboxes 16
# from your laptop, browse them:
inspect view --log-dir s3://my-evals-bucket/run-2026-06

Path 2 — Kubernetes for scale & isolation

For large agentic campaigns, the AISI inspect_k8s_sandbox provider runs each sample as a pod across the cluster, using Cilium Network Policies for fine-grained egress control. Point the task at the k8s sandbox and run as usual.

# install the provider, then select it from the task
pip install inspect-k8s-sandbox

# task.py
Task(..., sandbox="k8s")            # or sandbox=("k8s", "values.yaml")
# run against your configured cluster
inspect eval task.py --model openrouter/openai/gpt-5.4 --max-sandboxes 200

Path 3 — a managed platform

If you'd rather not operate Kubernetes, platforms like METR's Hawk run Inspect evals on cloud infrastructure from a YAML config — provisioning isolated pods, managing model credentials, streaming logs, storing results in a database, and serving a browse UI. This is the turnkey end of the spectrum (and conceptually what the Inspect Control Plane in the companion spec builds in-house).

Choosing — start on a VM; move to inspect_k8s_sandbox when you need many parallel sandboxes or strict per-pod egress; reach for a managed platform when you want the operational burden handled for you. Self-host on GPU nodes only when the model under test is open-weight.

Custom scorer that runs a unit test in the sandbox

@scorer(metrics=[accuracy(), stderr()])
def unit_test():
    async def score(state: TaskState, target: Target):
        r = await sandbox().exec(["python", "-m", "pytest", "-q"], timeout=120)
        return Score(value="C" if r.success else "I", explanation=r.stdout[-2000:])
    return score

17 Full example

End-to-end: a sandboxed coding agent

Putting every piece together — a coding eval where an agent fixes code inside a Docker sandbox and is scored by running the test suite. Three files plus one command.

task.py

from inspect_ai import Task, task
from inspect_ai.dataset import hf_dataset, Sample
from inspect_ai.agent import react
from inspect_ai.tool import bash, text_editor
from inspect_ai.scorer import scorer, accuracy, stderr, Score, Target
from inspect_ai.solver import TaskState
from inspect_ai.util import sandbox

@scorer(metrics=[accuracy(), stderr()])
def tests_pass():
    async def score(state: TaskState, target: Target):
        r = await sandbox().exec(["python", "-m", "pytest", "-q"])
        return Score(value="C" if r.success else "I", explanation=r.stdout)
    return score

@task
def fix_bugs():
    return Task(
        dataset=hf_dataset("my-org/bug-fixes", split="test",
                           sample_fields=lambda r: Sample(
                               input=r["task"], id=r["id"],
                               files={"/workspace": r["repo_dir"]})),
        solver=react(
            prompt="Fix the failing tests. Submit when green.",
            tools=[bash(timeout=180), text_editor()],
            attempts=3),
        scorer=tests_pass(),
        sandbox=("docker", "compose.yaml"),
    )

compose.yaml & Dockerfile

# compose.yaml
services:
  default:
    build: .
    init: true
    command: tail -f /dev/null
    cpus: 2.0
    mem_limit: 4gb
    network_mode: none

# Dockerfile
FROM python:3.12-slim
RUN pip install --no-cache-dir pytest
WORKDIR /workspace
# run it
inspect eval task.py --model openrouter/anthropic/claude-opus-4.8 \
  --epochs 3 --max-connections 10
inspect view
That's the whole shape — a dataset of repos with failing tests, an agent that edits and runs code inside a disposable Docker box, a scorer that runs pytest, repeated over epochs, with everything captured in the log. Swap the dataset, tools, and image and you have a different eval.
