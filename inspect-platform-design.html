at scale — observable, governed, FinOps-ready.

The goal: an open-source platform that anyone can deploy to run the full range of UK AISI Inspect evaluations — from lightweight model-graded jobs to heavy agentic, sandboxed jobs — without re-inventing the infrastructure each time. Everything is declared in Terraform and delivered through GitOps from a single repository, with logging, observability, and network inspectability wired in at both the account level and the per-run level, and GPU compute portable across AWS and external providers.

What this document is

This is the design specification for an open-source evaluation platform — free for any team to clone, deploy, and run. It covers what an eval actually is and how Inspect runs one, the infrastructure a simple workflow needs versus a complex one, the third-party dependencies it relies on, the capability layers and how each is provisioned in Terraform, how account-level and granular controls relate, how GPU compute stays portable across providers, and the GitOps flow that delivers all of it. It is written to be the architecture document that lives at the root of the repository.

// WORKLOAD

Inspect jobs

Model-graded and agentic/sandboxed evals from the inspect_evals collection, run as jobs rather than from a laptop.

// SUBSTRATE

IaC + GitOps

Every resource declared in Terraform, delivered through PR-gated pipelines. The control plane stays on AWS; GPU compute is portable across providers.

// SURFACE

One dashboard

A single console for passive inspection and active control, from account-wide health down to a single run's trace.

01 Evals 101

What an evaluation actually is

An evaluation measures a model's capability or safety on a specific task by running it against a dataset of examples and scoring the outputs. Instead of eyeballing a few prompts, you define the task once and run it reproducibly over hundreds or thousands of samples. Inspect, the open-source framework from the UK AI Safety Institute (AISI), is the tool this platform runs. It gives evals a common shape, so any eval — yours or one from the community — plugs into the same machinery.

The four building blocks of an Inspect task

// DATASET

Dataset & samples

The questions. Each sample is an input (the prompt) and usually a target (the correct answer or grading rubric).

  • Loaded from JSON/CSV files or straight from the Hugging Face Hub
  • Can carry metadata, expected tool use, and per-sample files
// SOLVER

Solver

How the model is driven. The simplest solver just calls generate(); richer ones add prompt templates, multi-step reasoning, or a full agent loop with tools.

  • Solvers compose — chain a system prompt, a tool-use loop, and a final answer step
  • This is where a "simple" eval becomes a "complex" agentic one
// SCORER

Scorer

How an output becomes a number. Scorers range from exact-match and pattern checks to model-graded rubrics where another model judges the answer.

  • Produces a score plus an explanation, per sample
  • Aggregated into metrics (accuracy, pass@k, …) across the dataset
// LOG

The .eval transcript

The authoritative record. Every prompt, completion, tool call, and score for every sample is captured in a single replayable .eval log.

  • Viewable in the Inspect log viewer; archived to S3 by this platform
  • Pinned to model version, git SHA, and seed for exact reproduction

Two classes of eval — and why the difference drives the whole platform

// LIGHTWEIGHT

Model-graded evals

The model answers a prompt; a scorer (often another model) grades it. No code execution, no sandbox. All the work is API calls and text. These are cheap, fast, and need almost no infrastructure — see the simple workflow next.

// HEAVYWEIGHT

Agentic, sandboxed evals

The model acts as an agent — running shell commands, editing files, browsing — inside an isolated sandbox. This needs container isolation, controlled network egress, audit, and often GPU compute. That's the complex workflow, and the reason the platform exists.

02 Simple workflow

How a simple eval runs, end to end

A model-graded eval is a straight line: load the dataset, prompt the model for each sample, score each completion, and write one .eval log. Nothing is sandboxed and nothing runs on a GPU — the model is reached over an API.

Defining and running it is just as direct — a task is a few lines, and one command runs it:

# task.py — a complete model-graded eval
from inspect_ai import Task, task
from inspect_ai.dataset import json_dataset
from inspect_ai.solver import generate
from inspect_ai.scorer import model_graded_fact

@task
def capital_cities():
    return Task(
        dataset=json_dataset("questions.jsonl"),
        solver=generate(),                 # just ask the model
        scorer=model_graded_fact(),        # a model grades the answer
    )

# run it from a laptop or a CI job:
inspect eval task.py --model openai/gpt-4o
03 Simple infrastructure

The infrastructure a simple workflow needs

Because nothing executes untrusted code, the footprint is tiny. You need somewhere to run the container, a secret for the model API key, and a bucket for the logs. No Kubernetes, no GPU, no NAT gateway. This is the platform's cheapest mode — and it runs for a few dollars a month.

04 Complex workflow

The infrastructure a complex workflow needs

Agentic evals let the model act: run shell commands, execute code, edit files, sometimes browse the web. That power is exactly why it must be contained. The model runs inside an isolated sandbox pod on Kubernetes, its network egress is allow-listed and recorded, and the model itself may be served from a GPU. Every one of these requirements maps to a piece of infrastructure the simple workflow didn't need.

Why this is the hard part — the jump from simple to complex is the jump from "trusted text" to "untrusted execution." Isolation, controlled egress, audit, and GPU serving are not optional extras; they are the price of letting a model act. The platform's job is to make that price a set of reusable modules instead of a bespoke build every time.

06 Capability layers

Logging, observability, and network inspectability

Each capability is provisioned as a Terraform module and exposed in the dashboard. The principle throughout: every layer answers a question at two altitudes — across the whole AWS account, and for one specific eval run.

// LOGGING

Structured, centralized logging

Every component emits structured JSON to a single sink, queryable by run, model, task, and sandbox type.

  • Control-plane, executor, and sandbox-pod logs to CloudWatch Logs, exported to S3 for cheap long-term retention
  • Each line tagged with run_id, task, model, git_sha
  • The .eval transcript itself is the authoritative per-sample record, archived to S3 with lifecycle rules
  • OpenSearch (optional) for full-text search across runs
// OBSERVABILITY

Metrics & traces

Operational health and eval health on the same panes.

  • Prometheus + Grafana for queue depth, throughput, error rate, per-run latency
  • DCGM exporter for GPU utilization, memory, power — only when GPU nodes are up
  • OpenTelemetry traces spanning submit → schedule → execute → score → archive
  • CloudWatch alarms on stuck queues, runaway runs, and budget thresholds
// NETWORK

Network inspectability

For agentic evals, what the agent's sandbox touched is a first-class, auditable record.

  • VPC Flow Logs on all subnets, to S3, partitioned by run
  • Kubernetes NetworkPolicy (Calico/Cilium) enforcing per-sandbox egress allow-lists
  • Cilium Hubble flow visibility for pod-level connection records during a run
  • Egress through a logged NAT / proxy so every outbound call from a sandbox is attributable
// IDENTITY

Audit & access

Who and what acted, end to end.

  • CloudTrail across the account, to a write-once S3 bucket
  • IRSA so each workload assumes a least-privilege role, no shared keys
  • Model API keys in Secrets Manager, never in images or state
  • Every dashboard active-action recorded as an attributable audit event
07 Two altitudes

Account level and granular level

The same capabilities are observable at the scope you need. The dashboard lets you start account-wide and drill to a single sample without changing tools.

ACCOUNT
Whole-estate view. CloudTrail, Config rules, GuardDuty findings, account-wide budget and Cost Explorer data, aggregate VPC Flow Logs, total runs and spend across every environment.
ENV
dev / prod workspace. Per-environment Terraform state, budgets, cluster health, executor queue depth, and spend, isolated by tag and account boundary.
RUN
A single eval run. Its logs, OTel trace, cost, model + git_sha + seed, and the exact sandbox network flows it generated — all keyed by run_id.
SAMPLE
One sample inside a run. The Inspect transcript, tool calls, sandbox exec records, and the egress connections that specific sample opened.
08 Cost reality

A hard look at what actually bills

Scale-to-zero handles the GPUs. The danger is the always-on tax — the components that bill 24/7 even on a month with zero evals. This is where eval platforms quietly leak money, so it gets named and engineered, not hidden. Figures are us-east-1 list prices, mid-2026, for a single environment; treat them as order-of-magnitude.

⚠ The EKS extended-support trap

The control-plane fee jumps from $0.10/hr ($73/mo) to $0.60/hr ($438/mo) — a 6× increase — once your Kubernetes version exits standard support, and Terraform-managed clusters drift into it silently because IaC defaults enable extended support. Pin the version with an upgrade-deadline comment in code so the intent survives review. That one comment prevents a ~$365/mo surprise.

Three structural levers that change the baseline

// LEVER 01

Ephemeral cluster

Create the EKS cluster per eval campaign and terraform destroy after. A cluster that lives 30h/mo costs ~$3 in control-plane fees vs $73 always-on. Best when runs are batched.

// LEVER 02

Kill the NAT tax

Route S3/ECR/CloudWatch/Secrets through VPC endpoints (S3 gateway endpoint is free) instead of NAT. For agentic egress, a small NAT instance can replace the gateway in dev — 90%+ cheaper.

// LEVER 03

Fargate-only mode

If a given period needs only model-graded evals, run with no cluster at all — the EKS module is opt-in. Baseline drops by the full $73 control-plane fee plus node overhead.

09 FinOps · opt-in tiers

Cheaper tiers for logging, monitoring & observability

Observability is the most over-provisioned layer in any platform — "turn everything on" defaults are what generate surprise bills. So each capability ships as a Terraform feature flag with a deliberately cheap default. You opt up into richer tiers, you never opt out of an expensive one.

Logging

TIER 0
default
S3-direct + CloudWatch Infrequent Access. Ship logs to the IA log class at $0.25/GB (half of Standard's $0.50) — it now supports full Logs Insights queries. Archive .eval transcripts straight to S3 with lifecycle-to-Glacier. ~$0 idle.
TIER 1
+ Standard CloudWatch for the few log groups needing metric filters / real-time alarms. Opt in per-group, not account-wide.
TIER 2
+ OpenSearch for full-text search across runs. Always-on and the priciest option — flagged off by default, justified only at high run volume.

Monitoring & metrics

TIER 0
default
CloudWatch metrics + a handful of alarms. Standard alarms are $0.10 each; basic AWS service metrics are free. Watch only what triggers action: stuck queue, runaway run, budget breach. No always-on backend. A few dollars/mo.
TIER 1
+ Amazon Managed Prometheus + Managed Grafana. Pay-per-ingest metrics; Grafana at $9/editor/mo. No node to babysit. The recommended middle tier.
TIER 2
Self-hosted Prometheus + Grafana on a small node (~$30/mo always-on). Only worth it if you want zero managed-service spend and accept operating it. Watch metric cardinality — it grows faster than budgets.

Network inspectability

TIER 0
default
VPC Flow Logs to S3 (cheaper than to CloudWatch) + Kubernetes NetworkPolicy enforcement. Captures the audit-grade record of what each sandbox touched at minimal cost. Near-$0 idle.
TIER 1
+ Cilium Hubble pod-level flow visibility, enabled only while a cluster is up — so it costs nothing in Fargate-only or torn-down periods.
TIER 2
+ live flow dashboards / deep retention. For investigation-heavy work; off by default.

The governing rule, expressed in Terraform

# Every capability is a gated, default-cheap feature flag.
variable "observability_tier" { default = 0 }   # 0 = CloudWatch only (cheap)
variable "logging_tier"       { default = 0 }   # 0 = S3 + CW Infrequent Access
variable "network_tier"       { default = 0 }   # 0 = Flow Logs + NetworkPolicy
variable "enable_eks"         { default = false }  # Fargate-only until you need sandboxes
variable "enable_opensearch"  { default = false }  # the expensive always-on extras stay off
variable "ephemeral_cluster"  { default = true }   # create per-campaign, destroy after

Defaults land you near the $141 baseline (or far below in Fargate-only mode). Richer tiers are a one-line change you make consciously — which is exactly the FinOps principle the project demonstrates: "the expensive switches exist, they're labelled, and they're off until someone decides otherwise."

11 FinOps in practice

Every run knows what it cost

The tiers above control the always-on spend. This layer makes variable spend attributable — each run carries its own cost, so spend is observable at the same altitudes as logs and traces.

  • Cost-allocation tags on every resource — run_id, backend, env, owner — so Cost Explorer slices spend by run and team.
  • Per-run cost computed from compute-seconds + token usage, stored beside the run metadata, visible the moment a run finishes.
  • Budgets & alarms at account and environment level, wired to dashboard alerts and optional auto-teardown.
  • Documented teardown — terraform destroy returns idle cost to ~$0, stated as an explicit platform feature.

Headline FinOps claim

The number to put in the README and the project's docs:

"Idle baseline ~$141/mo — or near $0 in Fargate-only / ephemeral mode. A lean active month runs ~$230, attributed per run, with the expensive observability switches off by default."

Quantified, tiered cost-awareness is what makes a platform safe for anyone to adopt. Measure it, tier it, then state it plainly.

12 GPU portability

Compute that isn't locked to one cloud

GPUs are scarce, priced unevenly across providers, and capacity comes and goes. The control plane, state, and observability stay on AWS — but GPU eval work is portable across AWS, RunPod, Lambda Cloud, and CoreWeave. A provider router picks where each job lands; the eval is identical wherever it runs because the Inspect sandbox contract is the same. (Azure is intentionally out of scope.)

// PRIMARY

AWS EKS — default

GPU node groups on spot, scale-to-zero via Karpenter. The default target: tightest integration with the control plane, IRSA, and network inspectability.

// K8S-NATIVE

CoreWeave

Kubernetes-first GPU cloud, often cheaper and broader on GPU types. Joins the same sandbox model as a managed cluster, so agentic evals run with minimal adaptation.

// BURST

RunPod

Fast, cheap on-demand and spot pods for burst capacity when AWS GPU quota is tight. Driven via its API/Terraform provider; best for parallel non-sandbox or container-sandboxed runs.

// RESERVED

Lambda Cloud

Cost-effective reserved GPU instances for predictable, sustained eval campaigns. Provisioned as remote Docker-sandbox hosts joined to the platform.

How portability is kept honest

Each provider is a Terraform module behind a common interface — gpu_provider selects the adapter, the rest of the spec is unchanged. The hard differences (K8s sandbox vs. Docker-sandbox-on-a-host, private networking, egress logging) are absorbed in the adapter so the eval definition never changes. Network inspectability is strongest on the K8s providers (AWS, CoreWeave) via NetworkPolicy + Hubble; on host-based providers (RunPod, Lambda) egress is captured at the host/proxy instead — a tradeoff the router records per run so the audit trail stays truthful about which guarantees applied.

# A job spec is provider-agnostic; the router resolves placement.
task:        inspect_evals/swe_bench_verified
model:       anthropic/claude-...        # or a self-hosted vLLM endpoint
sandbox:     k8s                         # none | docker | k8s
gpu:         { type: a100, count: 1 }
placement:   { provider: auto, max_usd_per_hr: 1.20, prefer: [aws, coreweave] }
# provider: auto → router picks cheapest backend meeting capacity + sandbox class
13 Provisioning

Everything in Terraform

No console clicks. Reusable modules, remote state with locking, separate environments. Observability, network, GPU providers, and the delivery pipeline are all modules — capability is provisioned, not bolted on.

14 CI/CD & GitOps

The repository is the control surface

Nothing reaches the platform except through Git. Infrastructure, eval definitions, and the dashboard all ship through the same reviewed, scanned, gated pipeline — so the live system always matches a known commit, and every change is auditable.

The GitOps loop

Two pipelines, one repo

// PIPELINE A

Infrastructure delivery

  • PR: terraform fmt/validate, tflint, checkov, and terraform plan posted to the PR
  • Policy gate: OPA/Conftest rejects cost or security anti-patterns (e.g. a public bucket, an un-pinned EKS version)
  • Merge → main: auto-apply to dev; prod requires a manual environment approval
  • Drift detection: scheduled plan; alert if the account no longer matches main
// PIPELINE B

Eval delivery

  • Eval specs live in the repo under eval-specs/, versioned and reviewed like code
  • PR label or merge triggers a run via the control-plane API
  • Run pinned to the spec's git_sha + seed + model version for exact replay
  • Results post back to the PR as a summary comment; dashboard links to the commit

Paved-path repository settings

The "paved path" is the easy, safe default: do the normal thing through these settings and you cannot push an unreviewed or unscanned change. The conventions are themselves codified (e.g. via a Terraform GitHub provider) so the repo's rules are reviewable too.

BRANCH
Protected main. No direct pushes; PR required; linear history. Required status checks: plan, tflint, checkov, policy gate — all must pass before merge.
REVIEW
≥1 approving review, stale approvals dismissed on new commits, and CODEOWNERS routing — infra/ to platform owners, eval-specs/ to eval owners.
IDENTITY
OIDC, no long-lived keys. GitHub Environments (dev, prod) hold the assume-role config; prod has required reviewers and a wait timer. Signed commits encouraged.
HYGIENE
Secret scanning + push protection on, Dependabot for actions and providers, pre-commit hooks (fmt, tflint, gitleaks) so problems are caught before CI.
PROVENANCE
Conventional commits + tagged releases. Each platform release is a tag; each run records the release it ran under, closing the loop from result back to source.

15 Build sequence

Phased plan

Built so it is demoable at the end of each phase. A finished narrow version always beats an unfinished broad one.

Phase 1 · Wk 1–2

Foundations + the easy path

Remote state, networking, storage, RDS, Secrets Manager, audit baseline (CloudTrail). Control-plane API + SQS. The CI pipeline (fmt, tflint, checkov, plan) and paved-path repo settings are in place from day one. Fargate executor runs non-sandbox API-model evals end to end; logs and .eval transcripts to S3; a first dashboard pane reads them. Demoable: open a PR, watch it plan and gate, merge, then submit a model-graded eval.

Phase 2 · Wk 3–4

The router + agentic path

EKS via Terraform; AISI k8s-sandbox configured with NetworkPolicy and Cilium/Hubble; run a sandboxed agentic eval from an in-cluster job. Router places jobs on Fargate vs. EKS by spec. Network inspectability pane goes live. Demoable: one platform, both eval types, with egress flows captured.

Phase 3 · Wk 5–6

Observability, FinOps, active control

Prometheus/Grafana + OTel traces; DCGM when GPU nodes are up. Tiered logging/monitoring flags, cost tags, budgets, per-run cost. Dashboard gains active actions (pause / kill / requeue / set-budget) with audit events. Scale-to-zero + spot. Demoable: the full console, account-to-sample, with the cost story.

Phase 4 · Stretch

Multi-provider GPU & deeper GitOps

Add the gpu-providers/ adapters beyond AWS — CoreWeave, RunPod, Lambda Cloud — behind the provider router, plus optional self-hosted vLLM and OpenSearch. PR-triggered eval runs with results link-back, and scheduled drift detection. Demoable: a job placed on the cheapest qualifying provider, end-to-end from a committed spec to a result posted on the PR.

Inspect Control Plane — platform design specification.

Scope boundaries, stated deliberately: no novel evals (use inspect_evals); no custom sandbox provider (use built-in Docker / K8s sandboxes); control plane and state on AWS, with GPU compute portable to RunPod, Lambda Cloud, and CoreWeave; Azure intentionally excluded. Stating limits keeps the project honest for the people who adopt it.
